💡 内部通信のセキュリティ強化は、障害時復旧能力・システム信頼性向上・コンプライアンスリスク低減につながる。スケーラブルな成長を支える基盤として、ユーザー体験の安定性向上に間接的に寄与。

サービス間のセキュアな内部通信

Service Bindingsを使用すると、あるVercelサービスから同じデプロイメント内の別のサービスへセキュアに呼び出すことが簡単になります。サービスが別のサービスのバインディングを宣言すると、Vercelは設定された環境変数を自動的に注入します。つまり、ユーザーコードは通常どおりそのURLをフェッチできますが、Vercelが内部的に書き換え、ルーティング、認証、TLSを処理します。これにより、Next.jsフロントエンドがFastAPIバックエンドを呼び出すようなVercel上のマルチサービスアプリケーションが可能になり、サービスの分離を保ちながらルーティング設定をシンプルに保つことができます:

```

{

services: {

my_frontend: {

root: frontend/,

framework: nextjs,

bindings: [

{

type: service,

service: my_backend,

format: url,

env: BACKEND_INTERNAL_URL

}

]

},

my_backend: {

root: backend/,

entrypoint: main:app

}

}

}

```

上記の例では、my_frontendはBACKEND_INTERNAL_URLへのfetch()リクエストを行うことでmy_backendサービスに到達できます。

```

export async function GET() {

const url = new URL(/users, process.env.BACKEND_INTERNAL_URL);

const res = await fetch(url);

const users = await res.json();

return Response.json(users);

}

```

フロントエンドは注入された内部URLを使用してバックエンドを呼び出します。トラフィックはVercelのネットワークを離れることはありません。FastAPI側では、my_backendが通常のルートを公開します:

```

from fastapi import FastAPI

app = FastAPI()

@app.get(/users)

def get_users():

return [...]

```

内部動作

そのURLはパブリックアドレスではなく、Vercel内部を指しています。コードがそれをフェッチすると、リクエストはVercelの内部ネットワークを経由して、デプロイメント内の他のすべてのリクエストを処理するのと同じルーティングレイヤーに移動し、そこからターゲットサービスに送られます。パブリックルートテーブルに到達することはなく、Vercelが TLSトラストをセットアップするため、証明書設定なしでプレーンなHTTPSフェッチが機能します。サービスは、パブリックな書き換えルートで公開するか、バインディングでプライベートに公開する場合にのみ到達可能です。

オブザーバビリティ

サービス間通信も可観測性に表示されるため、どのバインドサービスが呼び出されたか、およびリクエストにかかった時間を確認できます。

料金

サービス間通信はService RequestsおよびFast Origin Transferとして課金されます。CDN RequestsまたはFast Data Transferとして課金されることはありません。現在のレートについてはサービスの料金と制限を参照してください。
