あなたのデザインシステムは一人の判断で成り立っている—AIがそれを証明しようとしている

あなたのデザインシステムは一人の判断で成り立っている — AIがそれを証明しようとしている | Dolphiaによる | 2026年6月 | UX Collective

金曜日の午後6時40分。9件の貢献がレビュー待ちであり、その全てが同じ人を待っている。1件はボタンのホバー状態を変更する。別の1件は、見落とすと3つのチームを破滅させるようなやり方でトークンの名前を変更する。さらに別の1件は見た目は問題ないが、スクリーン リーダーを使用している人にはまったくアクセスできないデート ピッカーを追加しており、それが出荷される前に誰かが気付く唯一の方法は、私が今夜手作業でそれに気付くことだ。これは悪い金曜日ではない。これが仕事なのだ。

デザインシステムを所有すると、あらゆる変更がそこを通過する場所になり、長い間、私はそれが良いガバナンスだと思っていた：高い基準を気にかけている人によって保たれている。それが実際に何であるかを理解するのに、恥ずかしいほど長い時間がかかった。システムは単一の障害点を持っており、その障害点は私だった。

AIはそれを作ったわけではない。しかし今、それを無視することを不可能にしようとしている。

ゲートキーパーは常にリスクだった

最初の1年を超えて生き残ったあらゆるデザインシステムは、ゲートキーパーを開発する。時には人間で、時には小さなチームだが、機能は同じだ：誰かがトークンのコンプライアンス、命名の一貫性、コンポーネント API の正気さ、およびアクセシビリティをライブラリに入る前にレビューする。その人がシステムの一貫性を保つ理由だ。その人がシステムが停止する理由でもある。

ソフトウェアの世界でこれについての言葉がある。悲劇的なジョークから借りた言葉：バス係数。プロジェクトが深刻な問題に陥る前に、何人がバスに轢かれる必要があるか？多くのデザインシステムにとって、正直な答えは1人だ。

これは新しい観察ではなく、それを発見したかのように見せないよう気をつけたい。EightShapesのNathan Curtisは、ここ10年近く、デザイン システム チーム モデル、貢献、および「オーバーロード」パターンについて書いている。彼は、ほとんどの人がそれに遭遇する前の何年も前に、中央チームがボトルネックになるというダイナミクスに名前を付けた。当時の彼の結論は今でも正しいままだ：「前に進み続ける唯一の方法は、所有権を分散させることだ。」

難しい部分は、品質基準を分散させずに所有権を分散させることが、システムがどのように腐るかという方法だということだ。データは、ほとんどの人がこれを解決したことがないことを示している。Sparkboxのデザインシステム調査では、チームの61%が貢献プロセスを持っていたが、44%だけがガバナンス モデルを持っており、それが機能しているかどうかについて単一のメトリクスを追跡したのはわずか16%だった。Zeroheightのより最近のレポートは暗い：利害関係者の支持は前年比で42%から32%に低下し、Gartnerはデザインシステムをインフレした期待のピークから幻滅のどん底へスライドさせた後、そこに配置した。そしてそれを一緒に保つ人々は疲れている。2024年のTidelift保守者レポートでは、オープンソース保守者の60%が辞めたか、辞めることを検討しており、無給の保守者の61%は彼らのプロジェクトを単独で実行している。デザインシステムはオープンソース プロジェクトではないが、燃え尽きの形状は同じだ。品質保証と単一の障害点は、疲れ果てた人にとって同じだ。

ですから、AIが「デザインシステムを変革する」つもりだと誰かが言ったときは、どの部分かが最初の質問だ。なぜなら、痛む部分は、みんなが書いている部分ではないからだ。

ほぼ全員が間違ったレイヤーを解決している

現在のAIとデザインシステムに関する執筆の波を読むと、ほぼ全員が同じことについて書いていることに気づくだろう：AIがそれで構築できるようにあなたのデザイン システムを機械可読にすることだ。エージェントが解析できるトークン。LLMが消費できるメタデータを持つコンポーネント。プロンプトとして構造化されたドキュメント。「AIに対応したデザインシステム チェックリスト。」「エージェント デザインシステム」。前提は、デザイン システムはAIが読むことができるもので、その仕事はそれを読みやすくすることだ。その仕事は本物だ。それはまたボトルネックではない。

ボトルネックはレビューだ。ゲートであり、ライブラリではない。そして、ほぼ誰もゲートの自動化について書いていない理由は、それが不快な区別を強制するからだ。機械可読性の群衆はスキップできるものだ。

レビューは一つのものではない。それは同じコートを着ている2つのものだ。

強制がある：このトークンがバインドされているか、ハードコードされているか、このコントラスト比がパスするか、このコンポーネントのビジュアル スナップショットが変更されたか、APIは規約に一致するか？これらはルールだ。ルールは満たされるか、満たされないかのどちらかだ。
